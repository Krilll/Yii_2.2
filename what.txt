1. Тестирование

*При работе в виндосе (кроме консоли опенсервера)
используйте команды codecept.bat и yii_test.bat.
+ 1) Подготовка - создать и указать в тестовом конфиге БД
для тестирования, запустить yii_test migrate.
+ 2) Запустить тестирование находясь в корне проекта -
должны запуститься тесты в фронтэнде, бэкэнде и коммоне.
+ 3) Создайте класс юнит тестов во фронтэнде,
реализовав в нем проверки с помощью перечисленных методов,
 тест должен проходить успешно.
 Используйте хотя бы для одной проверки BDD стиль описания expect()->.
- $this->assertTrue - сравнении с true
- $this->assertEquals - равно ожидаемому значению
- $this->assertLessThan - меньше ожидаемого значения
- $this->assertAttributeEquals - значение атрибута (свойства)
объекта равно ожидаемому значению -
создайте экземпляр ContactForm,
заполните аттрибуты и проверьте,
можно так тестировать, например массовую загрузку значений атрибутов.
- $this->assertArrayHasKey - в массиве есть указанный ключ
+ 4) В фронтэнде создать класс функциональных тестов,
реализовать в нем проверку с помощью датапровайдера
содержимое активного (выделенного) пункта меню на всех страницах фронтэнда.
Тест должен проходить успешно

2. Создать и выполнить 3 миграции
Для task можно скопировать из предыдущего проекта и доработать.
1. В каждой миграции необходимо создавать
удаленные ключи по описанным связям между таблицами ,
например, чтобы создать ключ связывающий
поле field1 таблицы table1 и поле field2 таблицы table2 пишем в миграции
$this->addForeignKey('fx_table1_table2',
'table1', ['field1'], 'table2', ['field2'])
Если несколько связей между таблицами можно
добавить к названию ключа цифру, например fx_table1_table2_2
2. Не забываем реализовать в миграции откат,
проверьте как накатываются-откатываются миграции
3. Последовательность создания миграции -
как перечислены таблицы, иначе м.б. проблемы с ключами.
- yii migrate create create_new_table_task
таблица task, поля:

    =id - integer, primaryKey
    =title - varchar(255), not null
    =description - text, not null
    =project_id - integer, null, новое поле
    executor_id - integer, null, новое поле
    started_at - integer, null, новое поле
    completed_at - integer, null, новое поле
    creator_id - integer, not null
    updater_id - integer, null
    created_at - integer, not null
    updated_at - integer, null
    поля executor_id, creator_id и updater_id
    связаны с полем id таблицы user -
    три отдельные метода addForeignKey

таблица project, поля:

    =id - integer, primaryKey
    =title - varchar(255), not null
    =description - text, not null
    =active - boolean, not null, DEFAULT '0'
    =creator_id - integer, not null
    =updater_id - integer, null
    =created_at - integer, not null
    =updated_at - integer, null
    поля creator_id и updater_id
    связаны с полем id таблицы user

таблица project_user, поля:

    =id - integer, primaryKey
    =project_id - integer, not null
    =user_id - integer, not null
    role - enum ('manager', 'developer', 'tester'),
    спец. метода в миграциях для типа enum нет,
    описывается просто SQL кодом
    поля user_id связано с полем
    id таблицы user,
    project_id с полем id таблицы project
